# CLAUDE.md — CineMind Phase 1 Context for Claude Code

This file gives Claude Code everything it needs to help run, debug, and extend
Phase 1 of the CineMind movie recommendation system. Read this first.

Phase 1 no longer stands alone: `backend/` (FastAPI, Phase 2), `frontend/`
(React, Phase 3), and `streamlit_app/` (Phase 3.5) all now exist one level up,
at the project root, and all import Phase 1's artifacts and `src/` modules
directly. If you're touching those, also read the top-level `CLAUDE.md` (one
directory up) for the full-system view; this file stays scoped to Phase 1.

## Project goal
CineMind is a movie recommender combining a Two-Tower deep learning model with
content embeddings and an LLM reasoning layer. Phase 1 is the prototyping
phase: data loading, an RBM baseline, the two-tower model, vector search,
evaluation, and metadata enrichment. Reference repo the RBM baseline comes
from: github.com/nshakhapur/Movie_Recommendation_DeepLearning.

## Critical run order
Scripts MUST run in order 01 -> 06 (07 and 08 are independent, run any time
after 01). Each produces files/state the next consumes:
- 01 produces: data/train_positives.csv, data/test_positives.csv, data/items.csv
- 03 produces: artifacts/content_vecs.npy, artifacts/content_movie_ids.npy
- 04 produces: artifacts/two_tower.pt, artifacts/item_vecs.npy
- 05 produces: Qdrant collection 'collab_vectors' (or numpy fallback)
- 06 produces: artifacts/results.csv
- 07 produces: artifacts/movie_meta.csv (only depends on 01; needs OMDB_API_KEY)
- 08 produces: the Neo4j graph itself (only depends on 01; needs a running
  Neo4j -- `docker compose --env-file .env -f backend/docker-compose.yml up -d neo4j`)

## Data setup
- Download MovieLens 100K: https://files.grouplens.org/datasets/movielens/ml-100k.zip
- Unzip and place so that data/ml-100k/u.data exists.
- Data path is configured ONCE in src/cinemind_utils.py: DATA_DIR (default "data/ml-100k").
  Override with env var CINEMIND_DATA if needed.
- The raw files (u.data, u.item, u.user) are read by script 01 only.
  Scripts 02-06 read the cleaned CSVs that 01 creates.

## Verified results (MovieLens 100K, seed 42)
| Model | Precision@10 | Recall@10 |
|-------|-------------|-----------|
| Popularity baseline | 0.061 | 0.066 |
| RBM | 0.075 | 0.082 |
| Two-tower | 0.058 | 0.091 |
| Two-tower + pop prior | 0.093 | 0.126 |
These numbers are reproducible. If a change makes them much worse, something broke.

## Key technical decisions (do not "fix" these — they are intentional)
1. Temporal train/test split (not random): in cinemind_utils.temporal_split.
   For each user, last 20% of interactions by timestamp = test. Prevents future
   leakage. A random split would inflate metrics meaninglessly.
2. Implicit feedback: rating >= 4 counts as a positive; everything else ignored.
3. logQ correction in 04_two_tower.py: subtracts log(P(item)) from logits during
   training. WITHOUT THIS the two-tower loses to the popularity baseline. It is
   the single most important line. Do not remove it.
4. In-batch negatives + temperature 0.07: standard two-tower contrastive setup.
5. Hybrid scorer (two-tower + 0.30 * log_popularity): popularity is a feature,
   not just a rival. The hybrid beats both pure approaches.
6. E5 uses the SMALL model (e5-small-v2) for laptop speed. Production design
   calls for e5-large-v2; small is intentional for the prototype.

## Common tasks Claude Code may be asked to help with
- "It says FileNotFoundError" -> data not in data/ml-100k/, or script 01 not run.
- "Convert these to Jupyter notebooks" -> wrap each script's logical sections in
  notebook cells; keep cinemind_utils.py as an importable module.
- "Train on MovieLens 1M instead" -> 1M uses :: separators and different file
  names (ratings.dat, movies.dat). Update load_ratings/load_items in
  cinemind_utils.py accordingly; the rest is unchanged.
- "Speed up 04" -> reduce epochs from 60, or run on GPU if available (move
  tensors and model to cuda).
- "Add NDCG@10" -> extend cinemind_utils.precision_recall_at_k with a ranked
  gain calculation.

## Environment notes
- Python 3.10+. PyTorch CPU is fine; GPU optional.
- sentence-transformers needs internet on first run of script 03 only.
- Qdrant is optional; 05 falls back to numpy if no server on localhost:6333.
- Script 07 needs internet + an `OMDB_API_KEY` in `.env`; skip it entirely if
  you don't want poster/plot/cast enrichment -- nothing else depends on it.
- Script 08 needs a running Neo4j instance (`docker compose --env-file .env
  -f backend/docker-compose.yml up -d neo4j`); skip it if you don't want
  graph insights -- nothing else depends on it either.

## What NOT to do
- Do not commit data/ or artifacts/ to git (they are gitignored on purpose).
- Do not change the temporal split to random.
- Do not remove the logQ correction.
- Do not change the random seed (42) if comparing to the verified results above.

## Keeping System Architecture/02_project_phases.html in sync
`System Architecture/02_project_phases.html` is a status board of every notebook/file
across all project phases (1, 2, 3, 3.5), each tagged `done` / `drafted` / `planned`.
Whenever a session in this repo creates, completes, or runs a file that appears in
that table (a notebook in `notebooks/`, a script in `src/`, or eventually a file
under `backend/`, `frontend/src/`, or `streamlit_app/`), update that file's status
in `System Architecture/02_project_phases.html` in the same turn — do not wait to
be asked separately. Status rules, matching the legend already in the file:
- `done` — the file exists AND has produced its expected output/artifact (see the
  "Produces" column for Phase 1, or a successful run for later phases).
- `drafted` — the file exists on disk but hasn't been run end-to-end / has no
  artifacts yet.
- `planned` — the file does not exist yet.
Only edit the `<span class="pill ...">...</span>` status cell(s) and, if a new file
was added that isn't in the table yet, add a row for it — do not otherwise restyle
or restructure the diagram.

## Keeping System Architecture/01_system_architecture.html in sync
`01_system_architecture.html` is the 6-layer architecture diagram (L1 Data ...
L6 Frontend). Update it in the same turn whenever a change alters what it depicts:
- A new component, store, API route, or LangChain step is added, removed, or
  renamed (e.g. a new FastAPI route, a new Qdrant collection, Neo4j moving out of
  "later phase") -> add/edit/remove the matching `.comp` or `.step` block inside
  the correct `<section class="layer">` for that layer (L1-L6), and update its
  `.comp-desc` / `.step-body` text to stay accurate.
- The tech stack for a layer changes (new library, dropped dependency) -> update
  that layer's `.stackrow` tags.
- A layer's responsibility changes -> update its `.layer-role` one-liner.
Keep the color-per-layer convention (`--l1` ... `--l6`) and the legend at the top
consistent with any layer added or removed. Don't restyle unrelated layers.

## Keeping System Architecture/03_data_flow_diagram.html in sync
`03_data_flow_diagram.html` has three swimlanes: Lane A (offline batch), Lane B
(online request), Lane C (feedback/retraining loop). Update it in the same turn
whenever a change alters the actual sequence of data movement:
- A pipeline step is added, removed, reordered, or its behavior changes (e.g. a
  new caching step, a changed candidate count, a new cold-start branch) -> add,
  remove, or edit the corresponding `.pstep` block (and renumber the `.pnum`
  values that follow it) within the right `.pipeline` in Lane A/B/C.
- A new artifact is produced or consumed -> update the lane's `.lane-foot` note.
- The feedback loop itself changes (new signal type, new destination) -> update
  Lane C's steps and the `.loop-note` text.
Keep step numbering contiguous after edits (1, 2, 3, ...; sub-steps like 4a/4b are
fine when steps genuinely run in parallel) — do not leave gaps or duplicates.

## General rule for all three System Architecture diagrams
Treat `01_system_architecture.html`, `02_project_phases.html`, and
`03_data_flow_diagram.html` as living documentation, not one-off deliverables.
Before ending a turn that changed the pipeline, added/removed a file the diagrams
reference, or altered a data/control flow, check each of the three files above
against what actually changed and update whichever ones are now stale — don't wait
to be asked. If a change spans more than one diagram (e.g. a new backend route
affects both the architecture layer and the phases table), update all affected
diagrams together so they never drift out of sync with each other or with the
codebase.
